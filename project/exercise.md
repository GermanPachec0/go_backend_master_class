# Insert Customer

We can now use the code generated by sqlc.
Let's update the HTTP handler to insert the customer into the database.

We'll insert directly in the HTTP handler for now.
For simple applications, this is good enough.
As the project grows, we'll introduce a [repository](https://academy.threedots.tech/knowledge/repository-pattern) layer to separate database logic from the handler.

### sqlc queries

To use the code generated by sqlc, you need to create the *queries*:

```go
queries := dbmodels.New(db)
```

Then, you can call the proper method, like this:

```go
err = queries.InsertCustomer(ctx, dbmodels.InsertCustomerParams{
    // ...
})
```

The fields in the params struct will be passed to the SQL query placeholders (`$1`, `$2`, etc.).

### Dependency Injection

The handler needs a database connection to execute queries.
We could create a global variable or open a new connection inside each handler method, but **passing the database pool as a constructor parameter** is a better idea.
When you read `NewHandler(db)`, you immediately see what the handler needs to work.

```go
func NewHandler(db *pgxpool.Pool) Handler {
	if db == nil {
		panic("db cannot be nil")
	}

	return Handler{
		db: db,
	}
}
```

The import path for `pgxpool` is `github.com/jackc/pgx/v5/pgxpool`.

This pattern is called **[dependency injection](https://academy.threedots.tech/knowledge/dependency-injection)**. It sounds formal, but it means passing the `*pgxpool.Pool` as a parameter to the constructor instead of creating it inside. The dependency is explicit, and you control its lifecycle from one place.

Why not use a global variable? It's less explicit, so it's harder to understand what the handler depends on.
But the primary reason is testing. **Injecting dependencies makes it easy to swap out the real database connection for a mock or fake in tests.**

{{tip}}

This constructor injection pattern is the same one described in [Introducing Clean Architecture](https://threedots.tech/post/introducing-clean-architecture/).
In the article, dependencies are wired in `main.go`. In our project, `backend/orders/module.go`'s `Init` method plays the same role.

{{endtip}}

{{tip}}

We don't recommend DI frameworks like Wire, Dig, or Fx. Manual wiring is explicit and easy to debug. If your constructor list grows long enough to feel painful, that's not a problem with dependency injection. It's a signal that your dependency tree is too complex.

{{endtip}}

### Address Mapping

In {{exerciseLink "the previous exercise" "04-database" "02-generate-sqlc"}}, we configured sqlc so that shared types like `common.UUID` flow through both the HTTP and SQL layers without conversion.

But the OpenAPI-generated `Address` type and `shared.Address` are still separate structs. Even though they have the same fields today, **keeping them separate means either type can change independently.** The tradeoff is a small helper function to map between them:

```go
func openapiAddressToSharedAddress(addr Address) (shared.Address, error) {
	return shared.NewAddress(
		addr.Line1,
		addr.Line2,
		addr.PostalCode,
		addr.City,
		addr.CountryCode,
	)
}
```

{{tip}}

**Writing the boilerplate takes less time and effort than debugging mapping edge cases.** See [When to avoid DRY in Go](https://threedots.tech/post/things-to-know-about-dry/) for more on why duplicating structs across layers is often the right call.

{{endtip}}
